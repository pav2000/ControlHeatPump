Описание настроек даты и времени


MQTT
MQTT — это протокол обмена сообщениями по шаблону издатель-подписчик (pub/sub). Первоначальную версию в 1999 году опубликовали Энди Стэнфорд-Кларк из IBM и Арлен Ниппер из Cirrus Link. Они рассматривали MQTT как способ поддержания связи между машинами в сетях с ограниченной пропускной способностью или непредсказуемой связью. Одним из первых вариантов его использования было обеспечение контакта фрагментов нефтепровода друг с другом и с центральными звеньями через спутники.
С учётом суровых условий эксплуатации протокол сделан маленьким и лёгким. Он идеален для устройств слабой мощности и с ограниченным временем автономной работы. К их числу сейчас относятся и вездесущие смартфоны, и постоянно растущее число датчиков и подключённых устройств.
Таким образом, MQTT стал протоколом для потоковой передачи данных между устройствами с ограниченной мощностью CPU и/или временем автономной работы, а также для сетей с дорогой или низкой пропускной способностью, непредсказуемой стабильностью или высокой задержкой. Именно поэтому MQTT известен как идеальный транспорт для IoT. Он построен на протоколе TCP/IP, но есть ответвление MQTT-SN для работы по Bluetooth, UDP, ZigBee и в других сетях IoT, отличных от TCP/IP.
Система связи, построенная на MQTT, состоит из сервера-издателя (см. рисунок), сервера-брокера и одного или нескольких клиентов. Издатель не требует каких-либо настроек по количеству или расположению подписчиков, получающих сообщения. Кроме того, подписчикам не требуется настройка на конкретного издателя. В системе может быть несколько брокеров, распространяющих сообщения.
MQTT предоставляет способ создания иерархии каналов связи — своего рода ветвь с листьями. Всякий раз, когда у издателя есть новые данные для распространения среди клиентов, сообщение сопровождается примечанием контроля доставки. Клиенты более высокого уровня могут получать каждое сообщение, в то время как клиенты более низкого уровня могут получать сообщения, относящиеся только к одному или двум базовым каналам, «ответвляющимся» в нижней части иерархии.
Каналы гарантируют, что каждый клиент получает сообщения, предназначенные для него. Обрабатывая каналы как пути к файлам, MQTT выполняет все виды полезных функций связи, в том числе фильтрацию сообщений на основе того, где — на каком уровне или в какой ветви — клиенты подписываются на путь к файлу.

НК использует MQTT как канал передачи телеметриии ТН на сервер MQTT. Одним из достоинств использования MQTT в НК является его работа без белого IP адреса НК. т.е можно наблюдать за телеметрией НК через инет, при этом выход НК в сеть можно сделать через сотовую связь и обычный тариф (без 'плясок с бубном' VPN, DDNS и т.д.), которая заработает из коробки.

Настройка клиента MQTT - блок параметров отвечающих за настройку клиента для стандартного MQTT сервера, или за настройку клиента сервиса ThingSpeak. Надо понимать что данный блок параметров позволяет настроить ДВА варианта клиента (один из двух):
1. Стандартный клиент MQTT, через него можно соединяться с любым сервером MQTT, именно его надо использовать для интеграции с умным домом (Majordomo, OpenHAB, и. т.д.). Можно поднять соственный сервер, например Mosquitto. Передается TOUT, TIN, TBOILER, RCOMP или FC, ERROR, можно выбрать передачу дополнительных параметров (см. ниже).
2. Подключение к сервису ThingSpeak (https://thingspeak.com/) - бесплатный сервис (с некотороми ограничениями) для работы по протоколу MQTT, позволяет хранить данные в приличном объеме, есть масса клиентов для работы с этим сервисом на Андройде и IOS. Данные от НК пишутся в базу ThingSpeak, а затем их можно смотреть с телефона. Передается TOUT, TIN, TBOILER, RCOMP или FC, TCOMP, OWERHEAT, posEEV, ERROR. Набор параметров для передачи менять нельзя.
Основу сервиса составляю каналы, в которые и посылаются данные для хранения и визуализации. Каждый канал включает в себя 8 полей для любого типа данных, 3 поля для местоположения (широта, долгота, высота), и 1 поле состояния. Как только вы зарегистрируете в ThingSpeak свой канал вы сразу сможете отправлять туда данные, обрабатывать их и получать к ним доступ вашими приложениями. Каналы поддерживают JSON, XML и CSV форматы данных. Имеется возможность настройки приватного и публичного вида каналов, а также разнообразной обработки данных каналов, таких как усреднение и масштабирование. Можно также и читать данные из базы данных каналов используя, например, отметки времени, которыми получаемые данные снабжаются автоматически. Таким образом решается вопрос о размещении данных.
Касательно их визуализации, ThingSpeak предлагает разнообразные API для настройки визуализации. Ещё более продвинутые визуализации можно получить, используя разнообразные плагины и графические возможности, предоставляемые системой MatLab. При этом не требуется инсталлировать эту дорогую систему на собственном компьютере, т.к. все вычисления делаются в облаке. Достаточно написать на языке MatLab код как там обрабатывать Ваши данные и как их визуализировать. Для облегчения работы с системой на сайте ThingSpeak имеется обширная и толково написанная документация, снабжённая многочисленными примерами.
Включить передачу клиента MQTT - включение работы клиента MQTT. При этом передаются TOUT, TIN, TBOILER, RCOMP или FC, TCOMP, OWERHEAT, posEEV, ERROR.
Отсылка данных на ThingSpeak - включение данной опции - это настройка на на работу с сервисом ThingSpeak. если данная настройка не выбрана то работа со стандартным клиентом MQTT
Включить отсылку дополнительных данных - дополнтельно передается TCOMP, OWERHEAT, PCON, PEVA, EEV, powerCO, TCONING, TCONOUTG, TEVAOUTG, TEVAING. Только для стандартного клиента MQTT.
Включить передачу данных с электросчетчика SDM120 - дополнительно передается fullPOWER, CURRENT, VOLTAGE. Только для стандартного клиента MQTT.
Включить передачу данных об инверторе - дополнительно передается powerFC, freqFC, currentFC. Только для стандартного клиента MQTT.
Включить передачу данных об эффективности ТН - дополнительно передается COP, fullCOP. Только для стандартного клиента MQTT.
Интервал передачи данных (мин.) - интервал между посылками телеметрии, надо понимать что некоторые сервера имеют ограничения на минимальный интервал. Разумным интревалом является интервал 5-60 минут.
Адрес MQTT сервера - IP адрес где находится MQTT сервер. Можно вводить как IP адрес так и имя. Для сервиса ThingSpeak это mqtt.thingspeak.com
Порт MQTT сервера - порт для работы с MQTT сервером, обычно 1883.
ДляThingSpeak
Channel ID - регистрационные данные на сервисе ThingSpeak
Write API Keys - ключ для записи на сервисе ThingSpeak
Имя устройства на ThingSpeak - имя головного топика (устройства) на сервисе ThingSpeak
Для стандартного клиента MQTT
Логин для входа - логин для входа на сервер
Пароль для входа - пароль для входа на сервер
Идентификатор клиента - имя головного топика. Топик конкретного датчика Идентификатор_клиента/Имя_датчика

Передача данных на "Народный мониторинг" - блок парметров для работы сервисом "Народный мониторинг" (https://narodmon.ru/) через протокол MQTT. Это второй клиент НК для работы с MQTT сервером. Возможна параллельная работа первого клиента (стандартый MQTT или ThingSpeak) и клиента для "Народного мониторинга"
Народный мониторинг это основанный в 2011 году краудфандинговый геоинформационный проект по отображению на карте мира и контролю (на ПК, смартфонах и других гаджетах) показаний датчиков своих участников (температуры, влажности, атм.давления, скорости и направления ветра, радиации, энергопотребления и любых других величин), а также частных и городских веб-камер для публичного или приватного просмотра.
С декабря 2022 использование протокола MQTT стало платным и код был переписан для передачи через API народного мониторинга через TCP пакеты (Передача показаний по протоколам TCP/UDP на narodmon.ru:8283). При этом некоторые параметры оказались лишними и не используются в дальнейшем.
TCP также можно использовать для массовой загрузки показаний на сервер "задним числом" путем передачи в одном пакете данных показаний датчика в разные моменты времени, но не более 4КБ на пакет для TCP и 512 байт для UDP. Предпочтительно отправлять показания всех датчиков в одном пакете данных для минимизации издержек на инициализацию соединения с сервером и импорт показаний датчиков. Ответ сервера при успешной отправке будет OK (или текст ошибки или команда на исполнение), что рекомендуется использовать в качестве подтверждения получения данных, а при отсутствии ответа повторить отправку когда восстановится связь с сервером. При сообщениях об ошибках НЕ ПОВТОРЯТЬ! Конец строки ответа сервера завершается спецсимволом \n код 10(0A).
Включить передачу данных "Народный мониторинг" - включение клиента "Народного мониторинга", посылается TOUT, TIN, TBOILER, RCOMP или FC, ERROR
Посылать расширенный набор данных - к стандартным данным посылается еще TCOMP, OWERHEAT, PCON, PEVA, EEV, powerCO
Адрес сервера - IP адрес где находится MQTT сервер "Народного мониторинга". Можно вводить как IP адрес так и имя. Обычно narodmon.ru
Порт сервера (обычно 8283) - порт для работы по протоколу TCP/UDP, обычно 8283
Логин для входа - Не используется с декабря 2022 года, может быть любое значение.
Личный код для передачи - Не используется с декабря 2022 года, может быть любое значение.
Имя устройства (корень всех топиков) - имя устройства на сервисе
Справочно: Идентификатор на народном мониторинге - MAC адрес W5XX контроллера - надо понимать что инентификатором устройства на "Народном мониторинге" является MAC адрес. В НК MAC адрес задается программно в прошивке. По этому если скомпилировать прошивку из исходников без смены MAC адреса будет сутуация что масса устройств будут иметь одни mac адреса, и идентификаторы "Народном мониторинге" что не приемлемо. Поэтому отправка данных с адреса по умолчанию блокируется НК. Сменить MAC адрес можно в разделе СЕРВИС->СЕТЬ пункт 'Аппаратный mac адрес'
